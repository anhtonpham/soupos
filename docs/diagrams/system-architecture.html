APPLICATION · TANSTACK START ON VERCEL

One app: pages, typed API, and raw HTTP endpoints

A single TanStack Start app (React 19 + Vite, deployed as Vercel Functions via Nitro). Pages are server-rendered; browser code calls the server only through createServerFn RPCs whose inputs are validated with zod. Two endpoints that external services call (Stripe, cron) are plain server routes because they need raw request bodies.

DOMAIN LAYER · src/lib · THE BUSINESS RULES

The engine — one module owns every inventory decision

All correctness lives in inventory.ts. Pages, RPCs, the webhook, and the cron all call these same functions — there is exactly one code path that can hand out a seat, so there is exactly one place the no-oversell rule must hold (and it's covered by automated tests).

inventory.ts · 321 LINES

Inventory engine

  • 3reserveSeat — atomically claims 1 physical seat row (FOR UPDATE SKIP LOCKED), creates the pending order
  • markOrderPaid — idempotent pending→paid; honors hold, else re-claims, else refunds
  • createEvent / updateEvent — mints exactly ticket_limit seat rows; resize can only add/remove free rows
  • refundOrder / cancelHold — release seat back to free
  • verifyTicket / getEventStatus — read-only lookups; counts are derived from rows
payments.ts

Stripe adapter

  • 4createCheckoutSession — hosted checkout, client_reference_id = orderId
  • confirmCheckoutSession — retrieve session server-side, require payment_status='paid', then markOrderPaid
  • handleWebhookEvent — signature-verified backstop → same markOrderPaid
  • No-capacity edge → automatic refund
codes.ts · db/index.ts

Support

  • generateTicketCode — 128-bit random, URL-safe → the QR is unguessable
  • db client — Drizzle ORM over postgres.js, pooled, prepare:false (safe behind PlanetScale's transaction pooler)
  • Modes: PAYMENTS_MODE=stripe|free
SQL over TLS · every claim inside a single ACID transaction▼
DATA LAYER · PLANETSCALE POSTGRES · SINGLE SOURCE OF TRUTH

Four tables — and the inventory is physical rows, not a number

The key design choice: an event's capacity is represented by actual ticket rows pre-created when the drop is made. Selling means flipping a row's status — never inserting one. Availability shown to buyers is a derived count(*); it can render stale for a moment, but it can never sell wrong, because reads don't allocate — only the guarded claim does.

THE INVARIANT sold + held ≤ ticket_limit — always, physically. A 21st sale on a 20-seat drop would require a 21st ticket row, and one never exists. No counter to race, no counter to drift. 8 Buyer sees the QR pass; staff verify it read-only.
