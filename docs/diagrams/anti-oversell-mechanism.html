SOUPLEAF · HOTPOT PASS · THE ANTI-OVERSELL MECHANISM

Why this system physically cannot sell ticket #21

Most ticketing tools oversell because they check a number, then write a row — and under load, hundreds of buyers pass the check in the same instant. We removed the number entirely. Here is the exact mechanism, straight from the shipped code.

STEP 0 · THE BUG WE DESIGNED OUT

How normal systems oversell: the check-then-write race

A counter-based system asks is sold < 20? and then inserts a ticket. Between the check and the write there's a gap of a few milliseconds — and during a hyped drop, dozens of requests sit inside that gap at once. Every one of them saw "19 < 20". Every one of them inserts.

⚠ Scale that gap to a drop where hundreds click at once and you get 65 tickets sold against 20 — the app-level if statement can't stop it, because every request's check was true when it ran.
LAYER 1 · THE FOUNDATION

Capacity is physical rows — there is no counter to race

When a drop is created, the system mints exactly ticket_limit seat rows in the database (INSERT … FROM generate_series(1, 20)). Selling never inserts a ticket — it only flips an existing row from free to held to sold. A 21st sale would need a 21st row, and no code path in the entire system creates one after the drop is minted. Even the admin "resize" tool can only add rows or delete free ones — never below what's claimed.

LAYER 2 · THE CLAIM · reserveSeat()

Check and take are one indivisible database operation

When a buyer clicks Buy, everything below runs inside a single ACID transaction. The heart of it is one SQL statement in which finding a free seat and taking it are the same action — there is no gap for another buyer to slip into.

BEGIN transaction
sale window open? · per-person limit not exceeded? · buyer upserted by email
then▼
Create the order — status pending, expires in hold_minutes

THE CLAIM — one atomic statement
UPDATE tickets SET status='held', order_id=…, held_until=now()+hold
WHERE id = ( SELECT id FROM tickets
   WHERE event_id = $drop
     AND (status='free' OR (status='held' AND held_until < now()))
   ORDER BY id FOR UPDATE SKIP LOCKED LIMIT 1 )
RETURNING id;  -- a row back = yours · no row = sold out
▼
✓ A ROW CAME BACKSeat is held for this buyer. COMMIT, then send them to Stripe Checkout. The seat is theirs for the hold window — nobody else can touch it.
✗ NO ROW CAME BACKGenuinely sold out. ROLLBACK — the pending order evaporates, no Stripe session is created, nobody is charged. Buyer sees a clean "Sold out".
LAYER 2, UNDER FIRE · FOR UPDATE SKIP LOCKED

65 buyers hit "Buy" in the same second — what the database does

FOR UPDATE means a claimed row is locked until its transaction commits. SKIP LOCKED means each buyer's query skips rows other buyers are mid-claiming and grabs the next untouched one. So concurrent buyers don't queue up behind one row and don't collide — they spread across different seats. The moment all 20 rows are taken, the subquery finds nothing, and that is the sold-out signal.

LAYER 3 · THE HOLD LIFECYCLE

Abandoned checkouts release themselves — no cleanup job required

Look back at the claim query's WHERE clause: it accepts free rows or held rows whose held_until has passed. That means hold expiry isn't a scheduled job that could crash or lag — an expired hold is simply claimable by the next buyer, by definition. A cron job exists only to tidy up reporting labels; correctness never depends on it.

FREEMinted when the drop is created. Waiting to be claimed.
HELD · 10–15 MINBuyer clicked Buy and is on Stripe's payment page. Counts against availability so the drop can't over-promise.
SOLDPayment confirmed → 128-bit unguessable QR code issued. This is the Hotpot Pass.
…OR HOLD EXPIRESBuyer wandered off. The row's held_until passes and the very next claim can take it. Seat lost ≠ seat leaked.
↩ Refunds work the same way: an admin refund flips the seat back to FREE (and the QR code is wiped), so the seat is immediately resellable — still never exceeding the 20 rows that exist.
LAYER 4 · THE PAYMENT BOUNDARY · markOrderPaid()

Money can never mint a seat

Payment confirmation arrives twice by design — once from the buyer's success redirect (instant pass) and once from Stripe's signed webhook (backstop for closed tabs). Both call the same function, which first locks the order row so the two can't double-fulfill, then handles every case — including the nasty one where a buyer's hold expired but their payment still landed.

🔒 Gate: lock the order row (FOR UPDATE) → redirect and webhook are serialized; then branch:
CASE A · NORMAL (99%+)Order pending, seat still held → flip seat to SOLD, issue the QR code, mark order paid. Runs once; the second confirmation sees "already paid" and does nothing (idempotent).
CASE B · SLOW PAYERHold expired, seat was reclaimed by someone else → try to claim a different free seat using the exact same guarded query. If one exists, the buyer still gets a pass.
CASE C · SOLD OUT MEANWHILENo seat exists anymore → automatic Stripe refund, order marked refunded, buyer told plainly. The one thing that never happens: creating seat #21.
THE RECEIPTS

Not a promise — a property we test on every change

The exact disaster scenario is an automated test that must pass before any code ships (tests/concurrency.test.ts): seed a 20-seat drop, fire 65 truly concurrent buy requests at it, and assert exactly 20 succeed, 45 receive SOLD_OUT, and the database holds exactly 20 claimed rows. Companion tests prove expired holds free themselves (holds.test.ts) and that a late payment after sellout is refunded rather than oversold (payment-backstop.test.ts).

The invariant, in one sentence: a pass can only ever be one of the ticket_limit physical rows created with the drop — so sold + held ≤ limit isn't a rule the code enforces, it's a fact of how the data is shaped.
